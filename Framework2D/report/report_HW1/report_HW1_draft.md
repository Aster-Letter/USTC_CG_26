## 1. 项目概述

本次作业基于 Framework2D 中的 1_MiniDraw 模块展开。我在原始简单 Demo 的基础上，逐步将其扩展为一个支持多图元绘制、独立样式持久化、结构化界面、撤销重做、橡皮擦以及 PNG 导出的交互式绘图程序。

本项目最终实现的核心目标包括：

- 支持多种二维图元的交互绘制
- 支持线框样式与矢量填充
- 将界面重构为更清晰的三区布局
- 增强用户交互体验，包括撤销/重做、快捷键与橡皮擦
- 实现画布导出与导出目录选择
- 针对 Windows 中文路径与文件系统异常进行鲁棒性优化

目前已经实现的主要图元如下：

- 直线 Line
- 矩形 Rect
- 椭圆 Ellipse
- 多边形 Polygon
- 自由绘制 Freehand

实现的主要扩展功能如下：

- 独立样式快照机制
- 凹多边形填充修复
- 撤销/重做
- 快捷键支持
- 橡皮擦工具与视觉刷圈
- PNG 导出
- 导出目录选择

## 2. 系统架构与面向对象设计

### 2.1 总体结构

整个 MiniDraw 系统可以分为三个层次：

1. 图元层：由 `Shape` 抽象基类及其派生类组成，负责几何数据存储、图形绘制与命中检测。
2. 画布控制层：由 `Canvas` 负责处理鼠标事件、图元创建、历史记录管理与橡皮擦行为。
3. 窗口与界面层：由 `MiniDraw` 和 `Window` 负责组织顶部菜单、工具栏、属性栏、设置面板以及导出流程。

这一结构使得渲染逻辑、交互逻辑与界面逻辑彼此分离，便于迭代扩展。

### 2.2 Shape 抽象基类与多态机制

项目使用 `Shape` 作为统一抽象基类，各个具体图元都继承自该类，并实现统一接口：

- `draw(...)`
- `update(...)`
- `hit_test(...)`
- `add_control_point(...)`

这样一来，`Canvas` 在管理图元时无需区分具体类型，只需通过基类指针调用统一接口即可完成绘制与交互。这种设计体现了多态在图形对象系统中的典型应用。

简化后的思想如下：

```cpp
for (const auto& shape : shape_list_)
{
    shape->draw(config);
}
```

控制器不关心对象到底是 Line、Rect 还是 Polygon，只关心它是不是一个 Shape。

### 2.3 样式模型的重构

原始框架更接近“全局当前画笔样式”模型。如果仅保留全局状态，那么用户修改颜色或线宽后，历史图形在重新绘制时也可能跟着变化，这显然不符合真实绘图软件的行为。

因此，本项目对样式模型进行了重构：

- `Shape::Config` 中定义线色、线宽、填充开关与填充色
- 每个图元对象在创建时保存一份自己的 `Config` 快照
- 后续修改当前 UI 面板中的样式，只影响“下一次创建”的图元，不影响已经完成的历史图元

这一改动使渲染行为从“全局可变状态”转变为“对象自带样式状态”，更符合数据驱动渲染思想。

## 3. 核心交互与图元实现

### 3.1 拖拽式图元

以下图元使用“按下-拖拽-释放”的交互模式：

- Line
- Rect
- Ellipse
- Freehand

其共同特点是：

- 鼠标按下时记录起点并实例化图元
- 鼠标移动时持续更新几何参数
- 鼠标释放时若图元有效，则提交到历史列表

### 3.2 Ellipse 的包围盒实现

椭圆的实现不是简单把起点当作圆心，而是将拖拽的起点和终点作为包围盒对角点。由此实时计算：

- 圆心坐标
- 水平半轴长度
- 垂直半轴长度

该实现方式更符合用户直觉，也修复了早期椭圆定位偏移的问题。

### 3.3 Polygon 的点击式构建

Polygon 与拖拽式图元不同，采用的是“左键连续加点，右键结束”的交互方式：

- 第一次左键创建多边形对象
- 后续左键不断追加控制点
- 鼠标移动时最后一个预览点跟随鼠标更新
- 右键时执行 `finalize()`，移除预览点并闭合图形

这体现了不同图元在状态机设计上的差异。

### 3.4 Freehand 的连续采样

自由绘制模式下，系统在鼠标拖动过程中持续记录路径点序列，并在绘制时用折线连接这些点。为了避免过密采样导致数据量膨胀，代码中引入了最小采样距离阈值，从而在手感与效率之间取得平衡。

## 4. UI 架构重组与交互组织

### 4.1 三区布局

相较于原始零散按钮界面，本项目将 UI 重构为更清晰的工作区布局：

- 顶部菜单栏：Edit / File / Window
- 左侧工具栏：图元工具与橡皮擦工具选择
- 中央画布：核心绘制区域
- 右侧属性栏：当前样式与历史控制
- 设置窗口：全局设置与导出配置

这种布局更适合后续持续扩展，也使用户操作路径更明确。

### 4.2 工具切换与取消选择

工具栏支持以下工具：

- Line
- Rect
- Ellipse
- Polygon
- Freehand
- Eraser

在交互上，我额外实现了一个小优化：

- 第一次点击工具时激活该工具
- 第二次点击同一工具时取消选择，回到默认模式

这一细节虽然简单，但能明显提升使用体验。

### 4.3 快捷键支持

当前实现的快捷键包括：

- `Ctrl + Z`：撤销
- `Ctrl + Y`：重做
- `Ctrl + Shift + Z`：重做
- `Ctrl + E`：导出当前画布

这些快捷键通过窗口层统一监听，再分发给 `Canvas` 或导出逻辑。

## 5. 进阶功能与算法实现

### 5.1 基于快照的撤销/重做

若仅使用“删除最后一个图元”的方式实现撤销，则无法自然支持如下操作：

- 清空画布
- 橡皮擦删除中间图元

因此，本项目采用了快照式历史记录：

- `undo_history_`
- `redo_history_`

每次会修改画布图元集合的操作前，先保存当前图元列表快照。这样撤销/重做不再局限于“最后一个对象”，而是能够恢复整个画布状态。

### 5.2 命中检测与橡皮擦

为了实现橡皮擦功能，我在 `Shape` 基类中新增了：

```cpp
virtual bool hit_test(float x, float y, float tolerance) const = 0;
```

并在所有图元中实现各自的命中逻辑：

- Line / Freehand：点到线段距离
- Rect：边界距离，填充状态下可判断内部
- Ellipse：归一化椭圆方程
- Polygon：点到边距离与点在多边形内部测试

基于这一统一接口，橡皮擦能够删除当前鼠标位置下“最上层命中”的图元。

### 5.3 橡皮擦视觉反馈

为了增强交互可视性，橡皮擦模式增加了实时刷圈预览：

- 半透明填充圆表示有效擦除范围
- 外圈表示刷子边界
- 内圈与十字准星帮助精确定位

同时在工具栏和属性栏中都提供了橡皮擦半径滑条，便于动态调整刷子大小。

### 5.4 凹多边形填充修复

最初实现填充时使用了 `AddConvexPolyFilled`，因此在处理凹多边形时出现明显错误。后续将其替换为 `AddConcavePolyFilled`，成功修复了普通凹多边形的填充问题。

需要说明的是，当前仍未专门处理自交多边形，因此复杂拓扑情况不在本次实现范围内。

## 6. 工程挑战与鲁棒性优化

### 6.1 PNG 导出流水线

画布导出功能并非直接保存图元数据，而是基于当前 OpenGL 帧缓冲做截图导出。其流程如下：

1. 在 `Window` 基类中增加 `post_render()` 钩子
2. 在 ImGui 绘制完成后、交换缓冲前读取画布区域像素
3. 使用 `glReadPixels` 获取 RGBA 像素数据
4. 使用 `stbi_write_png` 写出 PNG 图像

该方案实现简单可靠，并且能准确反映用户最终看到的画布内容。

### 6.2 纯净导出

在最初版本中，如果用户在设置窗口中点击导出按钮，设置界面本身也会被一并截图到 PNG 中。

为了解决这个问题，我设计了“延迟一帧导出”策略：

- 用户点击导出时并不立即抓图
- 先记录导出请求
- 下一帧临时隐藏 Settings 与目录对话框
- 等到该帧渲染完毕后，再执行像素抓取

这样导出的就是干净画布，而不是“画布 + 面板叠加”的结果。

### 6.3 Windows 平台中文路径支持

在导出目录选择功能中，中文路径一度是最棘手的问题之一，主要表现为：

- 中文路径显示不正确
- 中文目录无法稳定进入或选择

围绕这个问题，项目进行了多轮排查与修复：

- 在 ImGui 初始化阶段加载 Windows 系统中文字体
- 在业务层引入 UTF-8 与 `std::filesystem::path` 的显式转换函数
- 调整文件对话框集成方式，使其走 `std::filesystem` 分支而不是窄字符回退分支

最终，中文路径显示与目录选择得到了明显改善。

### 6.4 控制台报错刷屏抑制

在使用 `ImGuiFileDialog` 浏览 OneDrive 目录时，控制台曾频繁出现如下错误：

```text
IGFD : last_write_time: The system cannot find the path specified.
```

问题本质不在“目录无法浏览”，而在于某些可枚举目录项无法稳定读取时间戳元数据。对此，我在底层库中将相关探测从抛异常接口改为 `error_code` 风格的非抛异常接口：

- 读取失败时仅清空日期/大小字段
- 不再将可恢复异常刷到终端

这属于典型的软件工程鲁棒性优化。

## 8. 总结与不足

本次 HW1 作业中，我不仅完成了基础图元绘制，还在原有框架之上持续扩展了较完整的交互式绘图系统，包括：

- 多图元支持
- 面向对象图元管理
- 独立样式持久化
- 填充功能
- 结构化 UI
- 撤销/重做
- 命中检测与橡皮擦
- PNG 导出
- 中文路径支持与文件系统鲁棒性优化

这些工作使得最终程序不再只是一个简单示例，而是一个具备较完整功能链路的小型 2D 绘图程序。

当前仍存在一些局限：

- 未专门处理自交多边形的复杂填充规则
- 导出依赖当前可见画布分辨率，而不是离屏高分辨率渲染
- 仅支持 PNG 导出，不支持更多格式
- 尚未实现对象选中与编辑，仅实现了橡皮擦作为拓展交互工具
- 文件对话框行为仍然依赖第三方库与 Windows 文件系统环境，部分极端目录结构可能仍需额外适配

总体而言，本次作业已经较完整地体现了图形交互程序中面向对象设计、事件控制、图元渲染、命中检测、历史记录与工程鲁棒性处理等多个方面的实践。
